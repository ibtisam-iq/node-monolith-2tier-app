## 1. What Kind of App Is This?

This is a **2-tier web application** built with:

| Layer | Technology | Location |
|---|---|---|
| **Tier 1 — Presentation + Application Logic** | React (frontend) + Node.js/Express (backend) | `client/` and `server/` |
| **Tier 2 — Data** | MySQL | `database/` |

The **Express server does everything** in one process — it serves the React frontend as static files **and** handles all API calls. There is no separate web server (like Nginx) in front of Express. This is what makes it a **monolith** — the frontend and backend run as a single combined Node.js process.

This project was later used as the base to build a proper **3-tier app** (in a separate repository: `node-3tier-app`), where the frontend, backend, and database each run as independent containers with proper separation.

## 3. Understanding the Architecture (2-Tier)

### Why This Is 2-Tier, Not 3-Tier

A **3-tier architecture** has three completely independent layers:
1. A dedicated **Presentation Tier** (e.g., React served by Nginx — has its own container/process)
2. A dedicated **Application/Logic Tier** (e.g., a Node.js API server — has its own container/process)
3. A dedicated **Data Tier** (e.g., MySQL — has its own container/process)

This project is **2-tier** because layers 1 and 2 are merged into a single process:

- The Express server in `server/server.js` does **both jobs**:
  - It **serves the React frontend** as static files via `express.static(path.join(__dirname, '../client/public'))`
  - It **handles all API requests** (`GET /api/users`, `POST /api/users`, etc.)

There is no separate frontend server. The browser talks directly to Express for everything — both for loading the UI and for making API calls. Only the database (MySQL) is a separate tier.

```
Browser
  │
  ▼
Express Server (Port 5000)
  ├── Serves  →  client/public/index.html  (the React app UI)
  ├── GET /api/users  →  queries MySQL  →  returns JSON
  ├── POST /api/users →  inserts into MySQL → returns JSON
  └── PUT/DELETE /api/users/:id → updates/deletes in MySQL
         │
         ▼
       MySQL (Port 3306)
```

### How the Two Tiers Work Together

1. The user opens `http://localhost:5000` in a browser.
2. Express serves `client/public/index.html` — this loads the React app.
3. React renders in the browser. When the user interacts (e.g., loads the user list), the frontend calls `GET /api/users`.
4. That request goes to the **same Express server** on the same port (5000).
5. Express queries MySQL, gets the user rows, and returns JSON.
6. React receives the JSON and renders the users on screen.

## 4. Deep Dive: The `client/` Directory (Frontend)

### `client/src/`

This is where **you write your React code**. It is the source — it never runs directly in the browser. It must be compiled first.

| File | Purpose |
|---|---|
| `index.js` | The React entry point. Mounts the `<App />` component into `<div id="root">` in `index.html`. |
| `App.js` | The root React component. Manages application-level state and renders child components. |
| `App.css` | Styles scoped to `App.js`. |
| `api/users.js` | Contains all HTTP call functions (`fetchUsers`, `createUser`, `updateUser`, `deleteUser`). These call `/api/users` on the Express backend. |
| `components/UserItem.js` | Renders a single user's data (name, email, role) in a row. |
| `components/UsersList.js` | Renders the full list of users by mapping over the data and rendering `UserItem` components. |

### `client/public/`

This folder has **two types of files** — it is important to understand the difference:

**Hand-written files (you created these, commit them to Git):**
- `index.html` — The HTML template for the app. Contains `<div id="root"></div>` where React mounts. It also has a `<script src="bundle.js">` tag that loads the compiled output.
- `style.css` — Static stylesheet for the overall page layout.

**Generated files (produced by `npm run build`, do NOT manually edit):**
- `bundle.js` — The compiled, bundled version of everything in `src/`. Webpack takes all your React components, JSX, CSS imports, and dependencies, and produces this single file. When someone opens the app in a browser, this is what actually runs.
- `bundle.js.LICENSE.txt` — Auto-generated by Webpack. Lists the open-source licenses of all third-party libraries included in `bundle.js`. You don't write this; don't delete it from production as it is required for license compliance.

> **Key insight:** In this project, `webpack.config.js` sets the build output path to `client/public/` — **not** to a `dist/` folder. So `bundle.js` lands directly in `public/`. That is by the developer's explicit choice in the config, not a default behavior.

### `client/webpack.config.js`

```js
const path = require('path');

module.exports = {
  mode: 'production',
  entry: './src/index.js',           // ← Start reading code from here
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'public'),  // ← Output goes to client/public/
  },
  module: {
    rules: [
      { test: /\.css$/, use: ['style-loader', 'css-loader'] },   // Handle CSS imports
      { test: /\.(png|jpe?g|gif)$/i, use: [{ loader: 'url-loader', options: { limit: 8192 } }] }, // Handle images
      { test: /\.js$/, exclude: /node_modules/, use: 'babel-loader' }, // Transpile JS via Babel
    ],
  },
  resolve: { extensions: ['.js', '.css', '.png'] },
};
```

**What Webpack does in plain language:**
1. Opens `src/index.js` as the entry point.
2. Follows every `import` and `require` statement to find all the code the app needs.
3. Runs each file through the appropriate loader (Babel for `.js`, css-loader for `.css`, url-loader for images).
4. Combines everything into a single `bundle.js` file.
5. Writes `bundle.js` into `client/public/`.

**This is a custom Webpack setup** — it is not Create React App (CRA). That means `webpack.config.js` is **mandatory**. Without it, Webpack does not know how to build the project.

### `client/.babelrc`

```json
{
  "presets": ["@babel/preset-env", "@babel/preset-react"]
}
```

Babel is a JavaScript compiler. Browsers cannot understand JSX (the HTML-like syntax in React components like `<UsersList />`). Babel converts it into plain JavaScript that browsers can run.

- **`@babel/preset-env`** — Converts modern JavaScript (ES6+, arrow functions, async/await, etc.) into older JavaScript compatible with more browsers.
- **`@babel/preset-react`** — Converts JSX syntax into `React.createElement()` calls.

**If you delete `.babelrc`:** The build will fail because Webpack uses `babel-loader` for `.js` files, and `babel-loader` needs Babel configuration to know what transformations to apply. It will not auto-regenerate.

### `client/package.json`

Defines the frontend's dependencies (React, ReactDOM, Webpack, Babel, etc.) and the build script. The important script is:

```bash
npm run build
```

This triggers Webpack, which reads `webpack.config.js` and produces `client/public/bundle.js`.

### The Build Output Folder — Where Does It Go?

This is determined entirely by `webpack.config.js`. The relevant line is:

```js
path: path.resolve(__dirname, 'public'),
```

`__dirname` here is the `client/` folder. So the output is `client/public/`. The developer made this choice deliberately. There is no magic — **wherever the developer writes the output path in the config, that is where the built files appear**.

In many other projects you will see `path: path.resolve(__dirname, 'dist')`. That is why a `dist/` folder is common. In those projects `dist/` is the build output. In this project, `public/` is the build output.

### What About the `dist/` Folder?

A `dist/` folder existed in an earlier version of this code when the project was first forked. That folder contained `bundle.js`, `bundle.js.LICENSE.txt`, and `index.html` — which means it was a Webpack build output from a previous config that used `dist/` as the output directory. It has since been deleted because the current `webpack.config.js` correctly outputs to `client/public/`, which is where Express looks for static files. **There is no `dist/` folder in the current codebase** and none should be created manually.

## 5. Deep Dive: The `server/` Directory (Backend)

### `server/server.js`

This is the **main entry point** of the entire backend. When you run `node server.js`, this file runs first.

```js
require('dotenv').config();           // Load .env file into process.env
const express = require('express');
const bodyParser = require('body-parser');
const mysql = require('mysql2');      // Note: mysql2, not the legacy mysql package
const cors = require('cors');
const path = require('path');

const app = express();
const port = process.env.PORT || 5000;

// Middleware
app.use(cors());
app.use(bodyParser.json());

// Database connection — credentials come from environment variables
const db = mysql.createConnection({
  host: process.env.DB_HOST,
  user: process.env.MYSQL_USER,
  password: process.env.MYSQL_PASSWORD,
  database: process.env.MYSQL_DATABASE,
});

db.connect((err) => {
  if (err) {
    console.error('Database connection failed:', err);
    process.exit(1);   // Crash the server if DB is unreachable
  }
  console.log('Database connected.');
});

// API Routes (inline in this file)
app.get('/api/users', (req, res) => { /* SELECT * FROM users */ });
app.post('/api/users', (req, res) => { /* INSERT INTO users */ });
app.put('/api/users/:id', (req, res) => { /* UPDATE users */ });
app.delete('/api/users/:id', (req, res) => { /* DELETE FROM users */ });

// Serve static frontend files from client/public/
app.use(express.static(path.join(__dirname, '../client/public')));

// For any route not matched above, serve index.html (SPA fallback)
app.get('*', (req, res) => {
  res.sendFile(path.join(__dirname, '../client/public', 'index.html'));
});

app.listen(port, () => {
  console.log(`Server is running on http://localhost:${port}`);
});
```

**Important observations from this file:**

1. **`mysql2` is used, not `mysql`** — `mysql2` supports MySQL 8's `caching_sha2_password` authentication. The older `mysql` package does not, which causes `ER_NOT_SUPPORTED_AUTH_MODE` errors with MySQL 8.
2. **All API routes are defined inline** in `server.js` — this is the monolith pattern.
3. **`express.static('../client/public')`** — Express serves the React app directly. No Nginx, no separate frontend server.
4. **The `*` wildcard route** — Catches all non-API URLs and serves `index.html`. This is the SPA (Single Page Application) fallback that allows React Router to work client-side.

### `server/app.js`

```js
const express = require('express');
const app = express();
const userRoutes = require('./routes/userRoutes');

app.use(express.json());
app.use(express.static('client/public'));

app.use('/api', userRoutes);

module.exports = app;
```

This file represents an **alternative, modular structure** for the Express app where routes are separated into a `routes/` folder and the app configuration is exported for use in `server.js`. In the current codebase, the primary entry point (`server.js`) defines its routes inline, while `app.js` is a parallel, refactored version. Both exist to show the two common patterns.

### Difference Between `server.js` and `app.js`

| Concern | `server.js` (current entry point) | `app.js` (modular pattern) |
|---|---|---|
| **DB connection** | Yes | No |
| **Routes defined** | Inline | Imports from `routes/` folder |
| **Starts the server** | Yes (`app.listen`) | No (only exports `app`) |
| **Purpose** | All-in-one runner | Pure app config, meant to be imported |

The pattern of separating `app.js` (configuration) from `server.js` (startup) is common in larger Node.js projects because it makes automated testing easier — you can `require('./app')` in tests without actually starting an HTTP server.

### `server/config/db.js`

Provides a reusable MySQL connection object using credentials from environment variables. This is imported by models or directly by `server.js` to run SQL queries.

```js
const mysql = require('mysql2');

const db = mysql.createConnection({
  host: process.env.DB_HOST,
  user: process.env.MYSQL_USER,
  password: process.env.MYSQL_PASSWORD,
  database: process.env.MYSQL_DATABASE,
});

module.exports = db;
```

### `server/controllers/userController.js`

Controllers receive an HTTP request (`req`) and send back a response (`res`). They contain the logic for **what to do** when a route is hit — in this case, calling the model to interact with the database.

```js
// Example — get all users
exports.getUsers = (req, res) => {
  User.findAll((err, users) => {
    if (err) return res.status(500).json({ message: 'Error fetching users', error: err });
    res.status(200).json(users);
  });
};
```

### `server/models/userModel.js`

Models contain the **SQL queries**. They know how to talk to the database. The controller calls a model function; the model runs the query and returns the result.

```js
const db = require('../config/db');

const User = {
  findAll: (callback) => {
    db.query('SELECT * FROM users', callback);
  },
  create: (userData, callback) => {
    db.query('INSERT INTO users (name, email, role) VALUES (?, ?, ?)',
      [userData.name, userData.email, userData.role], callback);
  },
};

module.exports = User;
```

### `server/routes/userRoutes.js` and `users.js`

Route files map **HTTP methods + URL paths** to the corresponding controller function.

```js
const express = require('express');
const router = express.Router();
const userController = require('../controllers/userController');

router.get('/', userController.getUsers);
router.post('/', userController.createUser);
router.put('/:id', userController.updateUser);
router.delete('/:id', userController.deleteUser);

module.exports = router;
```

Two route files exist (`userRoutes.js` and `users.js`) — they are different versions of the same routing logic created during development iterations.

## 6. Deep Dive: The `database/` Directory

### `database/init.sql`

```sql
CREATE DATABASE IF NOT EXISTS test_db;
USE test_db;

CREATE TABLE IF NOT EXISTS users (
  id    INT AUTO_INCREMENT PRIMARY KEY,
  name  VARCHAR(255) NOT NULL,
  email VARCHAR(255) NOT NULL UNIQUE,
  role  ENUM('Admin', 'User') NOT NULL
);
```

This SQL script:
- Creates the database `test_db` if it does not already exist.
- Creates a `users` table with four columns: `id`, `name`, `email`, `role`.
- Is automatically executed when MySQL starts inside Docker (by mounting it into `/docker-entrypoint-initdb.d/`).
- Can also be run manually against a local MySQL instance.

## 8. How the App Flows End-to-End

Here is the complete request lifecycle from browser to database and back:

```
1. Browser requests http://localhost:5000
        ↓
2. Express (server.js) receives request
        ↓
3. express.static middleware checks client/public/
   → Finds index.html → sends it to browser
        ↓
4. Browser downloads and parses index.html
   → Finds <script src="bundle.js"> → downloads bundle.js
        ↓
5. React app boots in browser
   → Calls GET /api/users on the same Express server
        ↓
6. Express routes the request to the /api/users handler
        ↓
7. Handler calls: db.query('SELECT * FROM users', callback)
        ↓
8. MySQL executes the query → returns rows
        ↓
9. Express sends rows as JSON back to browser
        ↓
10. React receives JSON → renders the user list on screen
```

## 9. Understanding Key Frontend Concepts

### What is Webpack?

Webpack is a **module bundler**. Modern JavaScript projects are split into many files (`App.js`, `UsersList.js`, `users.js`, etc.) and use `import`/`require` statements. Browsers cannot efficiently load hundreds of individual files. Webpack reads your entry point (`src/index.js`), follows all imports, and combines everything into one file (`bundle.js`). It also handles CSS, images, and transpilation via loaders.

### What is Babel / `.babelrc`?

Babel is a **JavaScript compiler**. React components are written in JSX, which is not valid JavaScript. Babel converts JSX like `<UsersList users={data} />` into `React.createElement(UsersList, { users: data })` that the browser understands. It also converts modern JS syntax (ES6+ features like arrow functions, destructuring, `async/await`) into ES5 that older browsers support.

Webpack calls Babel automatically for every `.js` file via the `babel-loader` rule in `webpack.config.js`.

### What is `bundle.js`?

`bundle.js` is the compiled, production-ready version of the entire frontend codebase. It is generated by running `npm run build` inside the `client/` directory. It contains all React components, their dependencies, CSS (injected at runtime), and third-party libraries, all in one minified file. This is the file the browser actually executes.

**`bundle.js` is a generated artifact — do not manually edit it. Always regenerate it by running `npm run build`.**

### What is `bundle.js.LICENSE.txt`?

When Webpack bundles your code, it includes code from open-source libraries (React, ReactDOM, etc.). Many of those libraries have license requirements that say their license text must be distributed with the code. Webpack automatically extracts those license comments and writes them into `bundle.js.LICENSE.txt`. This file is auto-generated alongside `bundle.js` every time you run a build. Do not delete it in production.

### What Files Should Be Committed to Git?

| File/Folder | Commit? | Reason |
|---|---|---|
| `client/src/` | ✅ Yes | Your source code |
| `client/public/index.html` | ✅ Yes | Hand-written HTML template |
| `client/public/style.css` | ✅ Yes | Hand-written stylesheet |
| `client/webpack.config.js` | ✅ Yes | Build configuration |
| `client/.babelrc` | ✅ Yes | Compiler configuration |
| `client/package.json` | ✅ Yes | Dependency manifest |
| `client/public/bundle.js` | ⚠️ Optional | Generated artifact — usually excluded; include only if you want a pre-built version in the repo |
| `client/public/bundle.js.LICENSE.txt` | ⚠️ Optional | Auto-generated alongside bundle.js |
| `client/node_modules/` | ❌ No | Installed packages — regenerated by `npm install` |
| `server/node_modules/` | ❌ No | Installed packages — regenerated by `npm install` |
| `.env` | ❌ No | Contains passwords — never commit |

## 10. Troubleshooting

| Error | Cause | Fix |
|---|---|---|
| `ER_NOT_SUPPORTED_AUTH_MODE` | Using the legacy `mysql` npm package with MySQL 8 | Switch to `mysql2`: `npm uninstall mysql && npm install mysql2` |
| `Database connection failed` on startup | MySQL not yet ready when backend starts | Add `depends_on` with health check, or add retry logic in `server.js` |
| `Cannot GET /` | `client/public/bundle.js` does not exist | Run `npm run build` inside `client/` first |
| `MODULE_NOT_FOUND` for any package | `node_modules/` not installed | Run `npm install` inside `server/` and `client/` |
| Frontend shows old data after code change | `bundle.js` not rebuilt | Run `npm run build` inside `client/` after any `src/` changes |
| MySQL data lost after container restart | No persistent volume defined | Add a named volume for `/var/lib/mysql` in `docker-compose.yml` |
| Backend cannot reach MySQL inside Docker | Using `DB_HOST=localhost` | Set `DB_HOST=db` (use the Docker Compose service name) |
